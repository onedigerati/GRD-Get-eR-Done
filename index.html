What Changed

GSD vs GRD

GRD isn't just a rename. It's a fundamental upgrade to how agents coordinate, communicate, and deliver results together.

Dimension GSD original GRD enhanced
Agent Model Fire-and-forget — agents spawn, execute, vanish. No awareness of siblings. Persistent teams — agents join named teams, see shared task boards, communicate in real-time.
Coordination Orchestrator polls each agent sequentially. No agent-to-agent channel. SendMessage enables direct agent-to-agent communication and broadcast alerts.
Task Tracking Implicit — orchestrator holds all state in its context window. Easy to lose progress. TaskCreate / TaskList / TaskUpdate — persistent shared board. Progress survives context resets.
Dependencies Hardcoded wave ordering. Orchestrator must manually sequence. blockedBy / blocks — declarative task dependencies. Agents auto-wait on predecessors.
Observability Orchestrator sees final output only. No intermediate visibility. Named agents + task board — see who's doing what, in real-time, across the whole team.
Lifecycle No formal lifecycle. Orphaned agents possible after failures. TeamCreate → work → shutdown → TeamDelete. Clean creation and teardown every time.
Failure Recovery Orchestrator must detect failure, re-spawn from scratch with repeated context. Team persists through failures. Idle agents can be re-messaged with new instructions without re-spawning.
Branding Informal naming. Not suitable for all workplaces. Professional and workplace-friendly. Same power, broader adoption.
Concrete Benefits

Why GRD is better

⚡
Faster execution
Persistent teams eliminate re-spawn overhead. Agents stay warm between waves, and the shared task board means zero wasted turns figuring out what to do next.
GSD: spawn agent → load context → work → discard → repeat
GRD: SendMessage to idle agent → resumes instantly
👁
Full visibility
Every agent has a name. Every task has a status. The team lead sees the entire operation at a glance through the shared task board, not hidden inside an orchestrator's context.
TaskList() → see all tasks, owners, blockers, and progress in one call
💬
Agent collaboration
Agents talk to each other, not just through the orchestrator. A researcher can flag a concern directly to the planner. An executor can warn siblings about a shared dependency.
SendMessage(recipient="planner", content="...") — direct peer communication
🛡
Resilient by design
If an agent fails, the team continues. Task state lives on the shared board, not in a single agent's memory. Re-assign work without replaying the entire workflow.
Failed task stays in_progress → reassign to new agent → picks up where it left off
📈
Smarter parallelism
Declarative dependencies replace manual wave ordering. The system knows which tasks can run in parallel and which must wait, without the orchestrator micromanaging.
addBlockedBy: ["task-1"] → task auto-starts when dependency completes
🏗
Clean resource management
Every team has a lifecycle. No orphaned agents, no leaked state, no zombie processes. Create, work, shutdown, delete — every time, automatically.
TeamCreate → work → shutdown_request → TeamDelete

What's New

Native-first modernization

GRD now leverages Claude Code's native capabilities instead of reimplementing them. Less custom code. Less token overhead. Same power — with the platform doing the heavy lifting.

⚙

Native model configuration SIMPLIFIED

/grd:set-profile removed. Each agent's .md file now has a model: frontmatter field that Claude Code reads natively. Edit the field directly — no custom profile resolution needed.

Before: Custom MODEL_PROFILES table, resolve-model command, /grd:set-profile workflow
After: model: claude-sonnet-4-5 in agent frontmatter. Edit directly. Done.
📋

Native todo tracking INTEGRATED

/grd:add-todo and /grd:check-todos removed. GRD workflows now use Claude Code's native TodoRead/TodoWrite tools to track deferred decisions, discovered improvements, and cross-phase items automatically.

Before: Custom todo system with file-based storage in .planning/todos/
After: Executor captures deferred items via TodoWrite. Pause, resume, progress, and transitions surface them via TodoRead.
💡

Auto-memory replaces manual tracking SIMPLIFIED

Four state commands removed (add-decision, add-blocker, resolve-blocker, record-session). Claude Code's auto-memory tracks decisions, blockers, and session context natively across sessions.

Before: Manual state add-decision calls after every important decision
After: Auto-memory persists decisions, blockers, and session context automatically
▶

Simplified session management SIMPLIFIED

Pause/resume workflows reduced from ~430 lines to ~60 lines. No more .continue-here.md handoff files. Auto-memory preserves full context between sessions. Pending deferred items surface automatically on pause and resume.

Before: 123-line pause workflow generating detailed handoff documents
After: Thin wrapper saves position to STATE.md, surfaces pending items. Auto-memory handles the rest.
📄

Leaner state tracking SIMPLIFIED

STATE.md template slimmed from ~100 to ~40 lines. Removed Decisions, Blockers, and Pending Todos sections — all now handled by auto-memory.

Before: ~100-line template with manual sections for decisions, blockers, todos, multi-field session info
After: ~40-line template: Project Reference, Current Position, Performance Metrics, single timestamp
🔄

Deferred work tracking NEW

The executor captures deferred decisions, out-of-scope improvements, and cross-phase dependencies as native todos. Other workflows surface them at the right moments — on pause, resume, progress checks, and phase transitions.

How: TodoWrite in executor. TodoRead in pause, resume, progress, and transition workflows.
Benefit: Deferred items survive context window boundaries with zero manual tracking.
Output Quality

Quality-of-output findings

Six differences between GSD and GRD that directly affect the correctness, reliability, and fidelity of generated artifacts. Each finding is traceable to specific source code evidence.

📋

Plan-checker catches user decision violations GRD Only

GRD plan-checker has a 7th verification dimension ("Context Compliance") that GSD plan-checker does not have. When a user makes decisions through /grd:discuss-phase, those decisions are recorded in CONTEXT.md. The plan-checker verifies that plans honor locked decisions, exclude deferred ideas, and don't contradict user preferences — before execution begins.

GSD plan-checker has 6 dimensions. It checks requirement coverage, task completeness, dependencies, key links, scope, and must_haves — but does not verify plans against user decisions.

Source: grd-plan-checker.md defines Dimension 7: Context Compliance. Checks: locked decisions have implementing tasks, no tasks contradict decisions, deferred ideas excluded. gsd-plan-checker.md has no equivalent dimension.
⚙

State updates are deterministic, not regex-based GRD Only

GSD agents update STATE.md by reading the file with cat, parsing with grep, and rewriting with manual string operations. If the markdown format deviates (extra whitespace, missing header, reordered sections), the regex can fail silently, producing corrupted state.

GRD agents use grd-tools.js state commands (advance-plan, update-progress, record-metric) that parse STATE.md with a structured parser and write back atomically. These operations have test coverage and handle edge cases.

Source: gsd-executor uses manual grep/sed STATE.md parsing. grd-executor uses grd-tools.js state commands. CHANGELOG v1.17.0: "Agents now use grd-tools for state updates instead of manual markdown parsing."
📈

Condensed prompts leave more context for actual work Improved

Both systems document the same quality degradation curve: 0-30% context = peak quality, 50-70% = degrading, 70%+ = poor. Agent instructions consume context before any task work begins. GRD executor instructions are ~48% smaller than GSD's, meaning more of the context window is available for reading code, writing files, and running commands.

For a task where the GSD executor starts at ~15% context usage (from its own prompt), the GRD executor starts at ~8%. This headroom keeps quality in the "good" range where GSD would enter "degrading."

Source: GSD executor ~785 lines vs GRD executor ~404 lines (48% reduction). GSD planner ~1387 lines vs GRD planner ~1158 lines (17% reduction). Both systems target 50% context per plan — GRD starts with more headroom.
📚

Two-step context assembly produces better-informed plans GRD Only

GSD planner reads project history by scanning all summary frontmatter with bash commands, then reading full summaries for phases it deems relevant. GRD planner uses grd-tools.js history-digest which compiles all summaries into structured JSON, scores each by relevance, selects the top 2-4, and reads only those in full.

The structured JSON is more reliable than bash-parsed YAML, uses fewer tokens, and produces better-contextualized plans because the planner makes more informed decisions about which prior work is relevant.

Source: gsd-planner uses bash frontmatter scanning across all summaries. grd-planner uses grd-tools.js history-digest with explicit scoring criteria. CHANGELOG v1.13.0: "grd-tools history-digest — compiles phase summaries into structured JSON."
✓

Executor self-check catches false completion claims GRD Only

GSD executor writes SUMMARY.md and proceeds to state updates. If a file creation failed silently or a commit was missed, the SUMMARY contains inaccurate claims that propagate to downstream phases.

GRD executor runs a self-check after writing SUMMARY.md: it verifies each claimed file exists on disk and each claimed commit hash exists in git log. If any check fails, it appends "Self-Check: FAILED" with missing items and does not proceed to state updates.

Source: grd-executor.md defines the <self_check> protocol: "Do NOT skip. Do NOT proceed to state updates if self-check fails." gsd-executor.md has no equivalent section.
🛠

Debugger commit operations are safer Improved

GSD debugger uses git add -A when committing fixes, which stages all changes in the working directory — including unrelated modifications or temporary artifacts. It checks commit_docs config using fragile bash grep patterns.

GRD debugger stages specific files individually (never git add -A) and uses grd-tools.js commit for planning docs, which automatically respects the commit_docs configuration. This prevents accidentally committing unrelated files.

Source: gsd-debugger uses git add -A and manual config check. grd-debugger stages files individually and uses grd-tools.js commit.

Quality gates at every stage

GRD introduces quality gates that GSD does not have: plan-checker context compliance (catches user decision violations before execution), executor self-check (catches false completion claims after execution), and plan structural validation (catches malformed plans before execution). Combined with deterministic state operations and reduced context consumption, these mechanisms reduce the probability of cascading errors across multi-phase projects.

Getting Started

Install GRD

One command. Works on Mac, Windows, and Linux. Supports Claude Code, OpenCode, and Gemini CLI.

Quick Install
$ npx get-er-done-cc@latest
Interactive installer — prompts for runtime selection and install location
1
Install
Run npx get-er-done-cc@latest and choose your runtime and install location.
2
Verify
Type /grd:help in your chosen runtime to confirm the installation.
3
Build
Run /grd:new-project to initialize your first project and start building.
Installation Methods
Recommended
Interactive (npx)
Run the installer and follow the prompts for runtime and location.
npx get-er-done-cc@latest
Non-interactive
Use flags to skip prompts. Ideal for Docker, CI pipelines, or scripted setups.
npx get-er-done-cc@latest --claude --global
npx get-er-done-cc@latest --all --global
Development (source)
Clone the repo for local development and testing modifications.
git clone https://github.com/glittercowboy/get-er-done.git
cd get-er-done
node bin/install.js --claude --local
Supported Runtimes
Claude Code
~/.claude/
npx get-er-done-cc@latest --claude --global
npx get-er-done-cc@latest --claude --local
OpenCode
~/.config/opencode/
npx get-er-done-cc@latest --opencode --global
Gemini CLI
~/.gemini/
npx get-er-done-cc@latest --gemini --global
CLI Flags Reference
Runtime
--claude Install for Claude Code
--opencode Install for OpenCode
--gemini Install for Gemini CLI
--all Install for all runtimes
Location
--global, -g All projects (home directory)
--local, -l Current project only (.claude/)
✓

Verify Installation

Open your runtime and run /grd:help to see all available commands. If the command is recognized, GRD is installed and ready to use.

💡
Recommended: Run Claude Code with claude --dangerously-skip-permissions for frictionless automation. GRD is designed for uninterrupted execution — stopping to approve git commit and date 50 times defeats the purpose. To stay current, update periodically with npx get-er-done-cc@latest.

Migrate from GSD to GRD

Already have a project built with GSD? One command migrates your .planning/ files. Agent Teams enhancements activate automatically.

GSD Project
Before migration
/gsd:plan-phase 3
gsd-planner, gsd-executor
~/.claude/get-shit-done/
branch: gsd/phase-03-auth
GSD ▸ PLANNING
→
migrate
GRD Project
After migration
/grd:plan-phase 3
grd-planner, grd-executor
~/.claude/get-er-done/
branch: grd/phase-03-auth
GRD ▸ PLANNING
Migration Steps
1
Install GRD

Install GRD globally. GSD and GRD can coexist — they install to different paths.

npx get-er-done-cc@latest --claude --global
2
Preview changes

Run the migration helper in dry-run mode to see what will change without modifying anything.

cd /your/project && node /path/to/get-er-done/bin/migrate.js --dry-run
3
Run migration
Auto-backup

Apply the migration. A full backup of .planning/ is created automatically before any file is modified.

node /path/to/get-er-done/bin/migrate.js # creates .planning-backup-{timestamp}/
4
Verify and continue

Run /grd:progress to see your project status through GRD. Pick up where you left off.

/grd:progress # shows current phase, next action, full context
What the Migration Helper Changes
References Updated
/gsd: → /grd: Command references in STATE, ROADMAP
gsd-* → grd-* Agent names in PLANs, SUMMARYs
GSD → GRD Branding in banners, labels
paths → get-er-done Tool path references in docs
Config & System
branch templates gsd/phase-* → grd/phase-*
brave API key ~/.gsd/ → ~/.grd/ (if exists)
auto-backup .planning-backup-{timestamp}/ created
verification Post-migration scan for remnants

Agent Teams activate automatically

You don't need to change anything in your project for Agent Teams to work. The enhancements live in GRD's workflow layer (the installed tool), not in your project's data layer (.planning/ files).

.planning/unchanged
STATE.md, ROADMAP.md, PLANs, config.json — your project data
↓ read by
GRD Workflowsenhanced
execute-phase.md, new-project.md, plan-phase.md, map-codebase.md — orchestration with Agent Teams
↓ spawns
Named Agentsteams
executor-01, stack-researcher, planner, arch-mapper — persistent teams, shared task board

The moment you switch from /gsd:execute-phase to /grd:execute-phase, your execution gets persistent teams, named agents, task boards, and SendMessage coordination — with zero changes to your existing .planning/ files.
